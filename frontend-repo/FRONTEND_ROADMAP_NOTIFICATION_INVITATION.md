# Frontend Roadmap: Invitations, Notifications, Mentions, SSE, and Thread Preferences

## 1. Title and Purpose
This roadmap defines the staged frontend work for the invitation, notification, mention/comment, SSE freshness, and thread-notification-preference enhancement.

It is a roadmap, not an implementation spec. The goal is to establish the build order and task boundaries so later per-task plan docs can be written cleanly and without re-litigating scope.

## 1.1 How To Use This Roadmap
- Implement tasks in order unless a later task is clearly pure hardening on top of an already stable surface.
- Write one implementation plan document per roadmap task.
- Do not merge multiple roadmap tasks into one plan unless the frontend codebase proves they are already inseparable.
- Treat the task names and numbering here as the stable reference for later planning work.

## 2. Diagnosis
- The frontend now has to treat invitations as versioned live state, not static rows.
- The inbox now has canonical read state, canonical unread counts, and multiple notification types that must stay distinct.
- SSE is only a freshness signal, so the frontend still needs stable REST flows before it can layer real-time invalidation on top.
- Comment notifications and mention notifications are separate rows, which means the inbox model and rendering logic must stop assuming one row can stand in for the other.
- Thread notification preferences are stored per thread and exposed as UI state, so the frontend has to read and write sparse preference state without promising unsupported delivery behavior.
- The backend change is large enough to require staged work because the same state model is reused across invitation lists, inbox rows, unread badges, mark-read actions, thread mention wiring, and thread preference UI. If those pieces are built out of order, the frontend will accumulate inconsistent local state and avoidable conflict handling bugs.

## 3. Guiding Policies
- REST canonical, SSE freshness only.
- One live invitation notification per invitation.
- Comment and mention are separate rows.
- Build vertical slices, no speculative UX behavior.
- Prefer boring integration over clever local state.

## 4. Roadmap Phases

### MVP
- **Task FE-01: Typed models and API client**. Goal: define the invitation, inbox, unread-count, SSE event, and thread-preference types plus the API wrappers needed by later surfaces. Why it exists: every later task depends on the same contract shape, so this is the safest place to lock the backend assumptions. Scope: typed request and response models, response unwrapping, error normalization, and endpoint wrappers for invitation lists, inbox lists, unread count, and thread preference reads and writes. Depends on: none. Done when: the frontend can compile against the new shapes without ad hoc parsing or local contract duplication.
- **Task FE-02: Owner invitation list surface**. Goal: render the workspace owner invitation list from the canonical invitation API. Why it exists: owners need a dedicated surface for invitation lifecycle management before those same rules are reused elsewhere. Scope: workspace owner invitation list, status filtering, pagination, pending versus terminal rendering, and owner-only action entry points. Depends on: FE-01. Done when: the owner can see pending and terminal invitations with the correct status-driven controls and pagination behavior.
- **Task FE-03: Invitee invitation list surface**. Goal: render the signed-in user's invitation list across workspaces. Why it exists: invitees need a dedicated view before invitation actions are reused inside inbox rows. Scope: current-user invitation list, status filtering, pagination, pending versus terminal rendering, and invitee-facing invitation summaries. Depends on: FE-01. Done when: the invitee can review their invitation inbox with the correct status labels and no invented action behavior.
- **Task FE-04: Owner invitation actions**. Goal: let owners edit role and cancel pending invitations from the owner list. Why it exists: invitation actions are versioned and conflict-prone, so the owner workflow needs to be proven on its own surface first. Scope: edit role, cancel, stale-version handling, terminal-state hiding, and row refetch after conflict or success. Depends on: FE-02. Done when: pending invitations can be updated or cancelled with server truth winning over stale local state.
- **Task FE-05: Invitee invitation actions**. Goal: let invitees accept or reject invitations from the invitee list. Why it exists: accept and reject carry versioned state and membership side effects, so the invitee workflow needs its own dedicated slice. Scope: accept, reject, version reuse, stale-version handling, terminal-state hiding, and workspace-context update after successful accept. Depends on: FE-03. Done when: invitees can resolve invitations and the UI refreshes from the returned invitation and membership state.
- **Task FE-06: Inbox list rendering**. Goal: render the canonical inbox with invitation, comment, and mention rows as distinct items. Why it exists: the inbox is the shared surface for notification state, so it must establish the row model before badge and mark-read logic is layered in. Scope: inbox list, row type rendering, invitation row action affordances, comment row presentation, mention row presentation, and pagination. Depends on: FE-01. Done when: the inbox shows distinct row types with the correct content and no local dedupe that hides separate comment and mention rows.

### Improvement 1
- **Task FE-07: Unread badge integration**. Goal: drive the header badge from the canonical unread-count source. Why it exists: the badge must not be inferred from visible rows because the backend unread total is authoritative. Scope: unread badge fetch, refresh wiring, and reconciliation after read actions. Depends on: FE-01, FE-06. Done when: the badge reflects the server unread count and updates without counting visible inbox rows.
- **Task FE-08: Single mark-read**. Goal: mark one inbox row read with server-backed state. Why it exists: single-row read is the smallest read-state action and the simplest way to validate canonical reconciliation. Scope: one-row mark-read action, local read-state update, rollback on failure, and badge refresh after success. Depends on: FE-06, FE-07. Done when: one row can transition to read and the badge reconciles with the server response.
- **Task FE-09: Batch mark-read**. Goal: mark multiple inbox rows read in one atomic action. Why it exists: batch read is a distinct server contract and must not be simulated with repeated single-row calls. Scope: multi-select or bulk action entry point, client-side duplicate prevention, atomic batch submit, and response-based unread-count reconciliation. Depends on: FE-06, FE-07. Done when: a batch read request updates only the rows the server confirms and the badge matches the returned unread count.
- **Task FE-10: Invitation actions from inbox rows**. Goal: let invitees accept or reject pending invitations directly from the inbox. Why it exists: inbox invitation rows are the primary place where the versioned invitation payload needs to be exercised in a notification context. Scope: row-level accept and reject actions, latest-version usage from the row payload, conflict refresh, and post-action row replacement or refetch. Depends on: FE-05, FE-06. Done when: invitation rows remain actionable only while pending and the row refreshes correctly after success or conflict.
- **Task FE-11: SSE client layer**. Goal: add one authenticated SSE connection that can parse the notification stream events. Why it exists: SSE is a freshness layer, so the frontend needs a clean event client before it can add invalidation behavior. Scope: connection lifecycle, snapshot handling, unread-count event handling, inbox invalidation event handling, and reconnect behavior. Depends on: FE-01. Done when: the frontend can open, keep alive, close, and reconnect the stream without leaking state.
- **Task FE-12: SSE inbox refresh behavior**. Goal: use SSE events to refresh inbox and badge state without replacing REST as source of truth. Why it exists: the stream only signals freshness, so the UI must translate events into refetches instead of treating them as authoritative row data. Scope: snapshot seeding, badge-only updates for unread-count events, inbox refetch on invalidation, and graceful fallback when SSE is unavailable. Depends on: FE-06, FE-07, FE-11. Done when: the inbox refreshes on invalidation, the badge updates on count events, and the app still behaves correctly if the stream drops.

### Improvement 2
- **Task FE-13: Mention picker wiring for thread create**. Goal: wire explicit mention selection into the thread-create flow. Why it exists: thread create now accepts mention IDs, and the frontend must send them intentionally rather than infer them from text. Scope: mention selection UI, selected-user payload wiring, limit and validation feedback, and thread-create submit integration. Depends on: FE-01. Done when: thread creation can submit explicit mention user IDs and the UI does not rely on free-text parsing.
- **Task FE-14: Mention picker wiring for thread reply**. Goal: wire explicit mention selection into the thread-reply flow. Why it exists: reply mentions share the same backend rules as create mentions but need their own surface integration. Scope: reply composer mention selection, selected-user payload wiring, limit and validation feedback, and reply submit integration. Depends on: FE-01, FE-13. Done when: thread replies can submit explicit mention user IDs and preserve the existing reply response shape.
- **Task FE-15: Thread notification preference read UI**. Goal: show the current per-thread notification preference state. Why it exists: users need a clear read path before the write path can be trusted. Scope: preference fetch, default-all display, per-thread mode labels, and read-state placement in the thread UI. Depends on: FE-01. Done when: the UI shows `all`, `mentions_only`, or `mute` correctly and treats absent preference as `all`.
- **Task FE-16: Thread notification preference write UI**. Goal: let the user update per-thread notification preference state. Why it exists: the preference is stored sparsely, so the frontend needs a write path that does not imply unsupported delivery semantics. Scope: mode selection, save action, optimistic or confirmed refresh, and handling `all` as the default state rather than a special durable mode. Depends on: FE-15. Done when: the selected mode persists correctly and the UI does not promise notification delivery suppression.

### Hardening
- **Task FE-17: Empty, loading, error, and conflict states**. Goal: standardize the state vocabulary across invitations, inbox, SSE, mentions, and preferences. Why it exists: the backend introduces stale-version conflicts, terminal states, and freshness invalidation, so the UI needs explicit recovery behavior. Scope: loading placeholders, empty states, retry states, conflict banners or refresh prompts, and stale-row handling across the new surfaces. Depends on: the relevant earlier tasks for each surface. Done when: each surface has a predictable response to loading, empty, error, and 409 conflict cases.
- **Task FE-18: Tests and QA hardening**. Goal: lock the roadmap work down with focused regression coverage. Why it exists: this feature spans multiple interconnected surfaces and is easy to regress if the behaviors are only checked manually. Scope: component tests, client contract tests, cross-surface interaction tests, and a short QA checklist for invitation, inbox, SSE, mention, and preference flows. Depends on: FE-01 through FE-17. Done when: the new frontend behavior is covered by automated tests and a repeatable manual verification pass.
